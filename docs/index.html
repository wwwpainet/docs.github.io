# 项目简介

项目基于 vue3，集成 vue3 最新生态系统的核心库实现， 主要集成 vite构建、 TypeScript、pinia、ESLint、Prettier、Stylelint、husky、lint-staged 、commitlint等技术栈

  • 框架： uni-app
  • 构建工具： Vite
  • 前端框架： Vue3.x
  • 编程语言： TypeScript
  • 代码规范：
    • ESLint
    • Prettier
    • Stylelint
  • 提交规范：
    • husky
    • lint-staged
    • commitlint
  • css 预处理器： scss
  • 状态管理工具：pinia

# 一、前端构建工具选择 vite 代替 webpack 提升前端开发体验

痛点：Vue CLI 基于 Webpack 构建工具生成，随着模块的不断增多，构建速度会越来越慢，开发体验上比较差

# 测试： vue3 项目不同构建工具对比

Dev 启动时长 Dev 启动加载 Build 时长
Vue cli 2568ms 320ms 5.14s
Vite 232ms 379ms 2.38s

测试两者的 dev 命令运行耗时相差十倍，且理论上，项目越大性能差距越大，最大的原因是 Vite 在开发模式下并没有做太多打包操作！

# 优势：

  • 1）Vite 更加轻量，并且构建速度足够快；

    Webpack 是使用 nodejs 去实现，而 Vite 使用 Esbuild 预构建依赖。Esbuild 使用 Go 编写，并且比以 JavaScript 编写的打包器预构建依赖快不是一个数量级；

  • 2）与 Vue 解耦，多框架支持, 对 TypeScript、JSX、CSS 等支持开箱即用；

  • 3）兼容 Rollup 插件机制与 API，方便拓展；

  • 4）Vue 官方推荐，对 vue 项目兼容性不错；

# 二、项目构建优化配置

  • 自动按需导入组件库以及自定义公共组件
  • 自动导入 vue 函数 如 ref, computed，watch 等，主要是为使用 vue3 Composition API 编码的时候不需要手动导入
  • 别名路径配置
  • pinia 数据持久化存储， 通过在 stroe 中配置 persist, 将会通过浏览器缓存来持久化存储数据.
  • 打包后移除 console 和注释
  • 打包压缩图片
  • 静态资源压缩，开启 gzip 压缩，生成压缩包.gz 文件
  • 分析构建模块包，通过配置 manualChunks  拆分代码，独立打包并按需加载
  • 打包分析,可视化分析构建包，查看模块占用空间大小以及模块的依赖关系
  • 监听配置文件修改自动重启 Vite

# 三、集成 Typescript 类型注解

  • 提升代码的可读性和可维护性：

    举个 🌰 看后端某个接口返回值，一般需要去 network 看 or 去看接口文档，才知道返回数据结构，而正确用了 ts 后，编辑器会提醒接口返回值的类型，相当实用。

  • 在编译阶段就发现大部分错误，避免了很多线上 bug

  • 增强了编辑器和 IDE 的功能，包括代码补全、接口提示、跳转到定义、重构等

# 四、集成 Pinia 状态管理器替代 vuex

Pinia.js 是 Vue 官方团队推荐代替 Vuex 的一款轻量级状态管理库。，因此也被认为是下一代的 Vuex，即 Vuex5.x，在 Vue3.0 的项目中使用也是备受推崇

# 优势：

  • 完整的 TypeScript 支持：与在 Vuex 中添加 TypeScript 相比，添加 TypeScript 更容易；

    不需要再创建自定义的复杂包装器来支持 TypeScript 所有内容都类型化，并且 API 的设计方式也尽可能的使用 TS 类型推断

  • 足够轻量，压缩后的体积只有 1.6kb;

  • 去除 mutations，只有 state，getters，actions（支持同步和异步）；

  • 使用相比 Vuex 更加方便，每个模块独立，更好的代码分割，没有模块嵌套，store 之间可以自由使用

    Pinia 通过设计提供扁平结构，就是说每个 store 都是互相独立的，谁也不属于谁，也就是扁平化了，更好的代码分割且没有命名空间。也可以通过在一个模块中导入另一个模块来隐式嵌套 store，甚至可以拥有 store 的循环依赖关系

  • 不需要注入、导入函数、调用它们，享受自动补全，让开发更加方便

  • 支持使用插件扩展 Pinia 功能

# 五、对 Storage 和 Map 封装的缓存进行整合

提高页面反应速度的一个方式就是使用缓存。缓存策略可以缩短页面请求资源的距离，减少延迟，并且由于缓存文件可以重复利用，还可以减少带宽，降低网络负荷, 因此对 Storage 和 Map 封装的缓存进行整合

  • Storage 和 Map 共用一套 api, 在命名上解决以下划线_开头命名的缓存到 Storage，并且 Map 也有副本

  • 不是以下划线_开头的表示会缓存到 Map 中，在程序生命周期内有并且在有效时间内有效

  • 可灵活设置缓存过期时间

  • 尽量不操作 Storage(读取速度慢)

# 六、 Request 全局网络请求封装

为了减少调用时的重复性代码以及进行一些全局配置, 方便我们统一处理后端返回的错误码，处理一些共同的逻辑以及更好的支持 Typescript，将 Request 网络请求基于 Promise 进行二次封装。

  • 将其封装为一个类，而不是一个函数的原因是因为类可以创建多个实例，适用范围更广，封装性更强一些。

  • 包含 config 常用请求配置项、封装请求拦截器、响应拦截器，请求方式配置，在发送网络请求前后，对请求头或响应结果等进行一些特殊处理的方法。

  • API 接口统一管理，单独划分出 API 层来管理项目的所有 API，以功能模块来划分每个 API 归属的文件。

  • 可以根据不同情况进行拓展

# 七、集成 Eslint、Prettier、Stylelint, 配置代码格式化检查和自动格式化

# 功能：

  • 在开发过程中保持代码风格统一以及代码质量检查。
  • 保存文件时自动校验、修复代码风格和质量
  • 格式化多种类型文件
  • CodeReview 时只需关注代码逻辑

# 特性：

eslint 侧重代码质量检查， 是做 js 的代码格式化的，但是扩展性很强，包括变量是否使用，是否要 console.log。 prettier 侧重代码格式化检查，支持多语言，并且只专注于格式化，范围更广。

vscode 的插件可以单独配置，单独起作用。但是如果项目根目录中有.eslintrc.js 和.prettierrc.js 文件，则以.eslintrc.js 和.prettierrc.js 的配置为标准，执行它们的配置。

prettier 的效果在 eslint 的效果之后，如果配置冲突了，则 prettier 会覆盖 eslint 的配置。

# 八、配置 Husky 和 Lint-staged 助力团队编码规范

  • husky + lint-staged 的渐进式方案,只检测 git 暂存区中的文件

  • lint-staged 是一个只检测 git 暂存区的 lint 工具，

  • husky 是一个用来给我们的项目添加 git hook 的工具，git hook 是进行 git 操作会触发的脚本，

例如：提交的时候会触发 pre-commit 钩子,输入提交信息会触发 commit-msg 钩子。 用 husky 安装 pre-commit 钩子，就可以在进行 git commit 操作的时候，运行脚本来检测待提交的代码是否规范，便可以只对暂存区的文件进行检查。

# 十一、Git代码提交规范

多人协作的项目中，在提交代码这个环节，也存在一种情况：不能保证每个人对提交信息的准确描述，因此会出现提交信息紊乱、风格不一致的情况。如果 git commit 的描述信息精准，在后期维护和 Bug 处理时会变得有据可查，项目开发周期内还可以根据规范的提交信息快速生成开发日志，从而方便我们追踪项目和把控进度。

# 规范 commit message 的好处:

  • 首行就是简洁实用的关键信息，方便在 git history 中快速浏览。
  • 具有更加详细的 body 和 footer，可以清晰的看出某次提交的目的和影响。
  • 可以通过 type 过滤出想要查找的信息，也可以通过关键字快速查找相关提交。
  • 可以直接从 commit 生成 change log。

# 十二、Vite上集成 module-federation（模块联邦）

通过vite和rollup提供的hock，对构建文件进行干预，从而将所有远程模块的组件，汇总到remoteEntry.js中。本地模块通过remoteEntry.js入口，从而调用加载三方组件以及组件编译后的js、css等文件。

  • 可以让我们在本地模块使用远程模块的纯js组件，用以抽取一些公共组件、团队开发一些相对独立的功能，完成独立部署并提供给其他本地模块使用。
  • 加载时只加载目标组件及其相关的内容
  • 使用时与本项目组件使用方式一致，因为它就是一个组件，只是远程加载而已
  • 允许一个线上部署的项目在运行时加载其他线上部署项目中的组件。
